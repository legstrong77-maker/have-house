## 資料流

1. **Scheduler** 在每月 2 / 12 / 22 日凌晨觸發
2. **ETL** 拉最新一季的 ZIP，解壓後對 22 個縣市的 a/b/c 三類 CSV 各自解析
3. 解析過程：
   - 跳過第二列（中文欄名說明）
   - 民國年 → 西元年；中文層次 → 數字
   - 計算單價/坪、屋齡
   - 偵測備註中的「親友、員工、債務、瑕疵、凶宅」等特殊交易，旗標 `is_special_deal`
4. 標準化地址後 upsert 到 `transactions`
5. 若 geocode_cache 有對應地址，附上 `geom`
6. REFRESH MATERIALIZED VIEW（區/月聚合、縣市總覽）

## 為什麼用 Materialized View

每筆查詢都重算 `percentile_cont` 對 10M 筆會慢。MV 在 ETL 後一次算好，查詢只要讀小表，FastAPI 端 50ms 內就回。

## 異常值處理

`mv_district_monthly` 和統計端點都加上：
- `is_special_deal = FALSE`
- `unit_price_per_ping BETWEEN 1000 AND 5000000`

避免「總價 1 元」、「車位算成住家」這類雜訊汙染統計。

## 隱私考量

- 實價登錄揭露的地址本來就是區段化（例 「OO 路 100~120 號」）
- 我們**不**做任何反向解析還原門牌；不留個資；只快取 normalize 後的地址 → 經緯度。

## 資料字典摘要

`transactions` 重要欄位：

| 欄位 | 意義 |
|---|---|
| `deal_kind`         | sale / presale / rent |
| `county_code`       | 'a'='臺北市', 'f'='新北市'... |
| `district`          | 鄉鎮市區 |
| `address_normalized`| 清理過的地址（保留尾號最低位） |
| `building_area_sqm` | 建物移轉總面積（平方公尺） |
| `unit_price_per_ping` | 單價/坪（= 單價/平方公尺 × 3.305785） |
| `total_price`       | 總價（元） |
| `is_special_deal`   | 親友、員工、債務、瑕疵等註記 |
| `age_years`         | 屋齡（年） |
| `geom`              | EPSG:4326 點位 |
